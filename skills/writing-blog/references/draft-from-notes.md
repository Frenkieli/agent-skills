# Draft From Notes

Use this reference when the user provides notes or rough source material and
wants a complete blog/article draft.

## Success Criteria

A draft succeeds when it has:

- a clear target reader
- a coherent central argument
- a structure that follows the reader's path of understanding
- Markdown frontmatter containing `audience`, `takeaway`, and `description`
- all meaningful author-provided material represented somewhere in the draft

## Workflow

1. Read the notes and extract the central argument, key cases, technical details,
   author judgments, and personal material.
2. Confirm writing intent from frontmatter and conversation context. Ask only for
   `audience` or `takeaway` when the missing answer changes the article
   direction.
3. Draft an outline using `structure-framework.md`. Show the outline first and
   wait for confirmation unless the user explicitly asks for a direct draft.
4. Write the article from the accepted outline. Put `audience`, `takeaway`, and
   `description` in frontmatter when producing a complete Markdown article.
5. Run a short self-check against the diagnosis checklist and mark only issues
   that remain relevant.

## Material Preservation

Do not delete author-provided material without reason. Personal experience, source of
inspiration, hesitation, emotional judgment, and rough examples may carry the
article's real intent.

When material does not fit the main argument, move it to a better position, fold
it into another paragraph, place it in background/origin material, or put it in
an appendix-like ending. Delete only when content is clearly repetitive,
contradictory, or outside the requested article, and explain that tradeoff.

## Draft Output

Output the complete article in Markdown.

At the end, include a short self-check note listing meaningful tradeoffs, such as
material moved out of the main flow, assumptions made from incomplete notes, or
places where author input would improve the article.
